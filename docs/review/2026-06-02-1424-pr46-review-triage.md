## Summary

CodeRabbit posted 7 unresolved actionable inline review threads. The findings
are mostly valid and relevant. The highest-risk issues are not Terraform HCL
shape, but operator safety around credential resolution: partial key pairs,
ambient AWS profile leakage, and preflight checks that can validate the wrong
credential source.

The PR goal remains correct:

```text
external k6 S3 writer credentials must be separated from the AWS/EKS shared stack
while preserving the existing AWS S3 artifact layout.
```

## Priority Map

### P1 - Must Fix Before Merge

1. `scripts/terraform-aws-s3-writer.sh`
   - Thread: https://github.com/Ahmad-mufied/monolith-vs-microservice-thesis/pull/46#discussion_r3339373379
   - Review: sourced env files may export `AWS_ACCESS_KEY_ID`,
     `AWS_SECRET_ACCESS_KEY`, or `AWS_SESSION_TOKEN`, causing Terraform to use
     env credentials instead of `TERRAFORM_AWS_PROFILE`.
   - Assessment: valid and important. This can silently apply the writer stack
     with the wrong AWS identity.
   - Fix plan: after sourcing env files, unset AWS key/session variables used
     for k6 writer credentials and export `AWS_PROFILE="$terraform_aws_profile"`
     before running Terraform.

2. `scripts/vultr-preflight-check.sh`
   - Thread: https://github.com/Ahmad-mufied/monolith-vs-microservice-thesis/pull/46#discussion_r3339373394
   - Review: AWS CLI checks run before verifying writer credentials and may use
     ambient AWS profile rather than the resolved k6 writer key pair.
   - Assessment: valid. Current order can hide the intended remediation and can
     pass with the wrong identity.
   - Fix plan: move credential guard before `aws sts`/`s3api head-bucket`, then
     run those AWS CLI checks with `AWS_ACCESS_KEY_ID`,
     `AWS_SECRET_ACCESS_KEY`, optional `AWS_SESSION_TOKEN`, and `AWS_REGION`
     pinned explicitly.

3. `scripts/lib/vultr-s3-credentials.sh`
   - Threads:
     - https://github.com/Ahmad-mufied/monolith-vs-microservice-thesis/pull/46#discussion_r3339373372
     - https://github.com/Ahmad-mufied/monolith-vs-microservice-thesis/pull/46#discussion_r3339373376
   - Review: loader can mix one env key with one Terraform key and silently
     falls back from `vultr_k6_s3_*` to `external_k6_s3_*`.
   - Assessment: valid. The output aliases currently point to the same resource,
     but relying on that fallback hides missing provider-specific outputs and
     creates a bad precedent for future provider-specific credentials.
   - Fix plan: treat env key pair atomically. If exactly one key is set, discard
     both and load both from `vultr_k6_s3_*`. Do not fallback to
     `external_k6_s3_*`. Fail fast with a clear error if the provider-specific
     pair cannot be loaded.

4. `scripts/lib/hetzner-s3-credentials.sh`
   - Thread: https://github.com/Ahmad-mufied/monolith-vs-microservice-thesis/pull/46#discussion_r3339373361
   - Review: loader falls back from `hetzner_k6_s3_*` to `external_k6_s3_*`.
   - Assessment: valid for maintainability and fail-fast behavior. The legacy
     shared fallback can remain for backward compatibility, but the new writer
     stack should require `hetzner_k6_s3_*` outputs, not generic aliases.
   - Fix plan: remove `external_k6_s3_*` fallback. Treat env pair atomically.
     Prefer `hetzner_k6_s3_*` from the new writer stack, then legacy
     `hetzner_k6_s3_*` from `infra/terraform/shared`, and fail if no complete
     pair exists.

### P2 - Should Fix

5. `Makefile`
   - Thread: https://github.com/Ahmad-mufied/monolith-vs-microservice-thesis/pull/46#discussion_r3339373321
   - Review: `aws-s3-writer-destroy-confirmed` does not run Terraform init
     before destroy.
   - Assessment: valid. It can fail in a fresh checkout or after `.terraform/`
     cleanup. It is not as dangerous as credential misuse, but it is a real
     reliability issue.
   - Fix plan: add the same `scripts/terraform-aws-s3-writer.sh init` call
     before destroy, preserving `TERRAFORM_AWS_PROFILE` and
     `S3_BENCHMARK_DATA_VERIFIED=true`.

## Invalid / Not Needed

No review thread is fully invalid. The only nuance is that `external_k6_s3_*`
currently aliases the same IAM user as `vultr_*` and `hetzner_*`, so it is not
immediately a different-principal bug. It is still relevant because it hides
missing provider-specific output wiring and weakens the separation contract.
