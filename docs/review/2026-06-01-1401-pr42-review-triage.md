## Priority Map

### P1 - Must Fix

#### 1. Secret values are exposed in `kubectl` argv

Thread:

- `scripts/create-eks-secrets-sequential.sh`

Why valid:

- The script currently creates secrets with repeated `--from-literal=...`
  arguments.
- Those values include `JWT_SECRET`, `DATABASE_URL`,
  `BOOTSTRAP_DATABASE_URL`, and benchmark admin credentials.
- These values can appear in process args while the command is running.

Current implementation evidence:

- `scripts/create-eks-secrets-sequential.sh:103-171`

Risk:

- security leak on operator machine
- possible exposure via `ps`, audit logging, or shell tracing

Recommended fix:

- switch to manifest generation via stdin with `stringData`, or
- generate `0600` temporary env files and use `--from-env-file`

Priority:

- `P1`

#### 2. Budget shutdown targets are hardcoded and can drift from configurable stack names

Thread:

- `infra/terraform/shared/main.tf`

Why valid:

- `cluster_names` and `rds_instance_ids` are hardcoded to `skripsi-*`
- the sequential naming path is now partly configurable
- this can make the shutdown Lambda target the wrong resources when names drift

Current implementation evidence:

- `infra/terraform/shared/main.tf:101-102`

Risk:

- budget guardrail misses actual active cluster/RDS
- cost leak during quota-sensitive benchmark work

Recommended fix:

- derive shutdown targets from the same source of truth as experiment naming, or
- intentionally make these names non-configurable and document that contract

Priority:

- `P1`

#### 3. Sequential recovery check ignores persisted env naming

Thread:

- `scripts/terraform-sequential-recovery-check.sh`

Why valid:

- the script computes `cluster_name` before loading `env/terraform.experiment.env`
- `env-init-eks.sh` now writes sequential naming data there
- custom names would make the recovery check inspect the wrong EKS/RDS pair

Current implementation evidence:

- `scripts/terraform-sequential-recovery-check.sh:4-9`

Risk:

- false clean recovery result
- missed drift detection
- operator confusion during interrupted apply recovery

Recommended fix:

- source `env/terraform.experiment.env` before deriving
  `SEQUENTIAL_CLUSTER_NAME`, `AWS_REGION`, and the RDS identifier

Priority:

- `P1`

#### 4. Sequential deploy script can race Job deletion and recreation

Thread:

- `scripts/deploy-sequential-architecture.sh`

Why valid:

- the script deletes fixed-name Jobs and immediately reapplies them
- Kubernetes Job deletion is asynchronous
- reruns can fail with `AlreadyExists` or immutable Job errors

Current implementation evidence:

- `scripts/deploy-sequential-architecture.sh:131-147`
- `scripts/deploy-sequential-architecture.sh:169-189`

Risk:

- flaky sequential redeploys
- brittle recovery on repeated deploy attempts

Recommended fix:

- add a reusable helper that deletes, waits for delete completion, then reapplies

Priority:

- `P1`

#### 5. Sequential single-run benchmark script can race benchmark Job recreation

Thread:

- `scripts/run-benchmark-sequential.sh`

Why valid:

- the script deletes `k6-benchmark-*` and immediately reapplies the same Job name
- no wait exists between delete and apply

Current implementation evidence:

- `scripts/run-benchmark-sequential.sh:154-157`

Risk:

- intermittent rerun failures
- wasted benchmark attempts

Recommended fix:

- wait until the old Job is fully gone before `kubectl apply`

Priority:

- `P1`

#### 6. Sequential suite script can race reset/seed/prepare Job recreation

Thread:

- `scripts/run-benchmark-suite-sequential.sh`

Why valid:

- the suite recreates the same Job names repeatedly inside one run
- there is no delete-wait between delete and apply

Current implementation evidence:

- `scripts/run-benchmark-suite-sequential.sh:180-207`

Risk:

- intermittent failures across suite cases
- especially likely after repeated reset/seed/enrichment operations

Recommended fix:

- add a shared wait-for-delete helper and use it for all fixed-name Jobs

Priority:

- `P1`
